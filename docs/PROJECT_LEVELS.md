# Project Levels

Open Tools Starter 使用 C / B / A 三档项目等级来控制复杂度。等级不是身份标签，而是交付边界：先把当前级别做完整，再考虑升级。

---

## C 级项目：轻量小工具

**适合**：单页工具、文本处理、提示词生成器、小型计算器、规则简单的小工具。

### 最低标准

| 类别 | 要求 |
|------|------|
| 核心功能 | 单核心功能，输入输出清晰，30 秒内可体验 |
| UI | 现代 UI，卡片式布局，圆角，柔和阴影 |
| 移动端 | 响应式布局，移动端不崩 |
| 深色模式 | CSS variables 或等价方案，刷新后保持 |
| 中英文 | UI 主路径文案完整，不混用半成品文案 |
| 示例数据 | 至少一组示例数据或示例入口 |
| 复制 | 一键复制可用 |
| localStorage | 可选（至少保存主题和语言） |
| README | 说明用途、运行、部署、隐私 |
| self-test | 可选 |
| GitHub Pages | 支持相对 base 部署 |
| 版本号 | 页面底部显示版本号 |
| 隐私 | 无后端、无上传、无登录 |

### 不强制包含

- PWA
- 复杂导入导出
- self-test.html / self-test.js
- preflight 脚本
- 分享链接
- 批处理
- RELEASE_NOTES

### 升级到 B 级的信号

当用户开始需要保存状态、导入导出数据、分享链接或更完整的错误处理时，考虑升级到 B 级。

---

## B 级项目：标准实用工具

**适合**：PDF 小工具、图片工具、文件处理、JSON/CSV 工具、有明确文件输入输出的实用工具。

### 最低标准

包含 C 级所有内容，额外必须：

| 类别 | 要求 |
|------|------|
| localStorage | 业务数据持久化封装 |
| 导入 JSON | 从本地 JSON 恢复配置或数据 |
| 导出 JSON | 把结果导出为 JSON 文件 |
| 下载结果 | 能保存处理结果为本地文件 |
| 分享链接 | 把当前状态编码到 URL |
| 错误提示 | 错误可理解、可恢复 |
| 空状态 | 不让用户面对空白页面 |
| self-test | self-test.html 和 self-test.js |
| RELEASE_NOTES | 每次发布记录变更 |
| 隐私说明 | 明确不上传用户文件 |
| 发布前检查 | preflight 或等价检查 |

### 控制原则

- 文件处理必须在浏览器本地完成，不上传到服务器。
- 不引入后端、数据库或登录系统。
- 所有按钮必须真实可用。

### 升级到 A 级的信号

当项目开始需要多模块、PWA、离线能力、批处理或统一组件库时，考虑升级到 A 级。

---

## A 级项目：旗舰项目

**适合**：长期维护的工具箱、多模块前端产品、旗舰级 GitHub Pages 应用。

### 最低标准

包含 B 级所有内容，额外必须：

| 类别 | 要求 |
|------|------|
| PWA | 支持安装和缓存 |
| 离线提示 | 提示当前离线状态 |
| 批处理 | 支持批量处理能力 |
| 模块化 | 工具可独立理解、独立测试、独立降级 |
| 组件库 | 统一可复用组件体系 |
| 设置中心 | 统一配置入口 |
| 健康检查 | 项目健康度检查脚本 |
| preflight | 发布前完整检查 |
| SEO/OpenGraph | 完整分享预览 |
| 错误边界 | ErrorBoundary 捕获运行时错误 |
| 中英文 | 完整双语文案 |
| 测试矩阵 | 完整测试覆盖 |

### 控制原则

- A 级不是"功能越多越好"，而是"长期维护能力完整"。
- 每次发布前必须运行 build、self-test 和 preflight。
- 不为炫技引入过重依赖。

---

## 不推荐行为（所有等级）

- 为了模板好看而做空按钮。
- 为了炫技引入重型依赖。
- 在 C 级项目中过早加入复杂批处理。
- 在纯前端工具里加入后端、数据库或登录。
- 未经明确说明上传用户文件。
- 把示例模块包装成核心产品功能。

## 升级建议

### 从 C 到 B

1. 补 localStorage、导入导出、下载结果文件和错误状态。
2. 补 self-test、RELEASE_NOTES 和隐私说明。
3. 补分享链接和发布前检查。

### 从 B 到 A

1. 做模块化和统一设置中心。
2. 补 PWA、离线提示、健康检查和 preflight。
3. 完善 SEO、OpenGraph、错误边界和完整双语文案。

## 质量检查

每个等级的项目都应通过对应的质量检查。详见 [docs/QUALITY_BAR.md](docs/QUALITY_BAR.md)。
